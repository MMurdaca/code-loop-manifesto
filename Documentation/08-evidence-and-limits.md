# Evidence And Limits

## Current Evidence

C.O.D.E. Loop has been tested through preliminary comparative pilots.

The basic method is simple:

1. run the same or comparable task with C.O.D.E. Loop available;
2. run it without the loop;
3. compare the outputs using a scorecard;
4. examine not only the final code, but also the rationale, risks, assumptions and validation evidence.

The early results suggest that C.O.D.E. Loop is strongest where governance matters.

It helps make visible:

1. what problem was being solved;
2. which constraints governed the work;
3. which assumptions were made;
4. why design choices were taken;
5. where risks or mismatches appeared;
6. whether implementation stayed aligned with upstream artifacts.

## What The Pilots Suggest

The pilots suggest that C.O.D.E. Loop can improve:

1. intent alignment;
2. constraint transparency;
3. governance discipline;
4. validation readiness;
5. rationale reconstructability;
6. risk and conflict surfacing.

These are exactly the dimensions the paradigm is designed to protect.

The benefit is not simply more output.

The benefit is more governable output.

## What The Pilots Do Not Prove

The current evidence is preliminary.

It does not prove that C.O.D.E. Loop always produces better software.

It does not prove that C.O.D.E. Loop is faster.

It does not prove that C.O.D.E. Loop always produces the most complete demo.

It does not prove statistical generality across teams, domains or agent systems.

This matters because the goal is not to create a mythology around the method.

The goal is to build a disciplined practice that can be tested, criticized and improved.

## Tradeoffs

C.O.D.E. Loop introduces friction.

That friction is intentional, but it must be proportional.

For small local changes, a full loop may be unnecessary.

For throwaway prototypes, the cost may exceed the value.

For high-risk changes, architectural work, multi-agent workflows, contract changes or long-lived systems, the cost of explicit artifacts may be far lower than the cost of silent drift.

The right question is not:

```text
Should every edit use the full loop?
```

The better question is:

```text
Does this change carry enough risk that intent, design and validation should be made explicit?
```

## Known Limits

C.O.D.E. Loop has limits.

It depends on the quality of the artifacts.

It depends on reviewers and agents taking validation seriously.

It can become bureaucracy if applied mechanically.

It can create a false sense of control if artifacts are produced but not actually used to govern decisions.

It does not eliminate engineering judgment.

It requires it.

## Open Questions

Several questions remain open:

1. how lightweight the smallest useful loop can be;
2. which gates can be automated safely;
3. how much authority agents should have in approving downstream propagation;
4. how to integrate the model with IDEs, issue trackers and CI systems;
5. how to evaluate outcomes across larger teams and longer projects.

These questions are part of the work.

C.O.D.E. Loop should evolve through evidence, not slogans.
